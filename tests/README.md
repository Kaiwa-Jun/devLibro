# 開発手法：ゆるいTDD + CDD（Comment Driven Development）

このプロジェクトでは、**ゆるいTDD（Test Driven Development）**と**CDD（Comment Driven Development）**を組み合わせた開発手法を採用しています。これは、厳密なTDDの運用が難しい場合でも、テストと設計のメリットを活かしつつ、柔軟に開発を進めるためのアプローチです。

## 🧪 ゆるいTDDとは？

TDDは、以下のサイクルで開発を進める手法です：

1. **Red**：失敗するテストを書く
2. **Green**：テストを通す最小限の実装を行う
3. **Refactor**：コードの重複を排除し、可読性を高める

「ゆるいTDD」では、このサイクルを厳密に守るのではなく、状況に応じて柔軟に適用します。例えば、テストを書く前に簡単な実装を行ったり、テストと実装を並行して進めることも許容します。

## 💬 CDD（Comment Driven Development）とは？

CDDは、**コメントを先に書く**ことで、コードの設計や実装の方向性を明確にする手法です。具体的には、以下のようなコメントをコード内に記述します：

- 関数やクラスの目的
- 入力と出力の仕様
- エッジケースや例外処理の方針

これにより、実装前に設計の意図を共有しやすくなり、チーム内でのコミュニケーションが円滑になります。

## 🔄 開発フロー

1. **仕様の確認**：実装する機能の要件や目的をIssueとして明確にします。
2. **コメントの記述（CDD）**：コード内に、関数やクラスの目的、仕様、処理の流れなどをコメントとして記述します。
3. **テストの作成（TDD）**：コメントを基に、期待する動作を確認するためのテストコードを作成します。
4. **実装**：テストを通すための最小限の実装を行います。
5. **リファクタリング**：必要に応じて、コードの構造を改善し、可読性や再利用性を高めます。

## ✅ メリット

- **設計の明確化**：コメントにより、実装前に設計の意図を明確にできます。
- **コミュニケーションの促進**：コメントを通じて、チーム内での情報共有が容易になります。
- **柔軟な開発**：TDDのサイクルを厳密に守る必要がないため、状況に応じて柔軟に開発を進められます。
- **品質の向上**：テストを通じて、バグの早期発見や仕様の確認が可能になります。

## ⚠️ 注意点

- **コメントの更新**：実装内容が変更された場合、コメントも適切に更新する必要があります。
- **テストの網羅性**：柔軟なTDDの適用により、テストの網羅性が低下しないよう注意が必要です。
- **チーム内の合意**：この手法を採用する際は、チーム内での合意と理解が重要です。
